Key insight: send() returns immediately — it does NOT block for a network call. The record lands in the in-memory RecordAccumulator (per-partition ring buffer). A background Sender I/O thread drains batches and does the actual TCP write. This is why the producer is async by default — and why sendAndWait in the MQ client must explicitly call Future.get() to block until the broker callback fires.

Tuning rule: High-volume producer → bump linger.ms to 5–20ms + compression.type=lz4 + bigger batch.size. Fewer, larger, compressed batches = dramatically less network I/O. Trade-off: ~10ms extra latency per record (usually acceptable for async flows). linger.ms=0 is correct for low-latency, low-volume use cases like UMS onboarding events.

Config Values Default What it does
acks 0 / 1 / all all (3.0+) durability 0=no ack (fire & forget, fastest, data loss risk) · 1=leader ack (UMS current) · all=all ISR replicas ack (strongest, required for idempotence)
enable.idempotence true / false true (3.0+) reliability Assigns PID + per-partition sequence numbers. Broker deduplicates retries and rejects out-of-order sequences. Exactly-once per partition per producer session. Forces acks=all, retries>0, max.in.flight≤5.
batch.size bytes 16384 (16KB) throughput Max bytes per batch per partition. Sender flushes when batch hits this OR linger.ms expires. Larger = fewer, bigger network writes.
linger.ms milliseconds 0 throughput How long to wait accumulating records before flushing, even if batch isn't full. 0 = send ASAP (low latency). 5–20ms = better batching (high throughput).
compression.type none / lz4 / zstd / snappy / gzip none throughput Compresses the batch before sending. lz4=fastest CPU, good ratio. zstd=best ratio, moderate CPU. Applied per-batch — only useful with batching (linger.ms > 0).
max.in.flight.requests
.per.connection		 integer 5 reliability Max unacknowledged batches in-flight to one broker. With idempotence, broker tracks state for ≤5 batches to guarantee ordering. Old advice: set to 1 for ordering without idempotence — now obsolete.
retries integer MAX_INT (3.0+) reliability How many times to retry a failed send. With idempotence, retries are safe (broker deduplicates). Without idempotence, retries can produce duplicates.
delivery.timeout.ms milliseconds 120000 (2 min) reliability Total time budget for a record (from send() to ack or failure), including retries. Caps retry loops. Must be ≥ request.timeout.ms + linger.ms.
UMS current profile (from MQ control plane): acks=1 · enable.idempotence depends on MQ server config · acksConfig field in ProducerDetails is set at producer init time, not per-send. To change: update via MQ platform team, not application.yml.
